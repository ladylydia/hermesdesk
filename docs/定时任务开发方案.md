## 一、定时任务在 Kabuqina 当前是"半残"的

按你定的方向去看上游 + 现有代码，发现单加 messaging 还不够，定时任务这条链路还有几个隐藏断点必须一起补：

| 环节                                        | 上游有没有                                              | Kabuqina 现在状态                                                                   | 必须补                     |
| ----------------------------------------- | -------------------------------------------------- | --------------------------------------------------------------------------------- | ----------------------- |
| 1. `cronjob` tool（Agent 创建/查/改/删定时任务）     | ✅ 有，在 `cronjob` toolset                            | ❌ 没在 KEEP_LIST — 用户问"明早 8 点提醒我"，agent 根本看不到这个工具                                   | 加 toolset               |
| 2. `messaging` tool（cron 任务触发后把结果发出去）     | ✅ 有 `send_message_tool`                            | ❌ 没在 KEEP_LIST                                                                    | 加 toolset               |
| 3. Cron scheduler tick（每 60 秒检查一次哪些任务到点了） | ✅ `cron/scheduler.py:tick()`                       | ⚠️ 只在 gateway child 里跑 (`gateway/run.py:_start_cron_ticker`)，不开 gateway 的用户根本不会触发 | 必须在 web child 也跑一个      |
| 4. 本地交付（"提醒"如何送达桌面用户）                     | ❌ 上游 cron 必须有 `delivery_target` 比如 `telegram:#foo` | ❌ 完全没有                                                                            | 新增 "desktop" 平台作为本地交付通道 |
| 5. 审批桥                                    | ✅ 有 shell 命令的 `approval_bridge`                    | ⚠️ 只覆盖 shell，没有 messaging/cron                                                    | 复用结构，扩展到 messaging      |

第 3 条和第 4 条是上游不会替你解决的——必须 Kabuqina 自己做。

## 三、`tool_policy.py` 改动方案

# 当前

KEEP_LIST = ["web", "file", "vision", "image_gen", "tts", "skills", "todo", "browser"]

POWER_USER_EXTRA = ["terminal", "code_execution", "moa"]

# 改为

KEEP_LIST = [

"web", "file", "vision", "image_gen", "tts", "skills", "todo", "browser",

"cronjob", # 新增：定时任务（核心需求）

"messaging", # 新增：跨渠道发送（cron 交付 + 主动通知）

]

POWER_USER_EXTRA = ["terminal", "code_execution", "moa"]

# messaging 不放 EXTRA — 标准用户也要有

不需要给 cronjob/messaging 单独搞 power_user 分级，靠审批策略来区分行为：

- Power user: 审批桥短路，全部放行
- Standard user: 命中审批策略的调用走 Tauri 弹窗

## 四、审批策略（核心设计点）

关键判断：Cron 不能在"任务执行时"才弹审批。 因为 8 点触发时用户可能在睡觉/不在电脑前，弹窗等不到响应。所以审批模型必须分两种时机：

### 4.1 创建时审批（Cron 任务）

用户在 /chat 说"每天 8 点把今天日程发到我 Telegram"，agent 调用 `cronjob(action="create", ...)`：

[Standard User]

agent → cronjob create

→ 拦截，弹 Tauri 审批对话框

→ "AI 想创建一个定时任务：

- 触发：每天 08:00

- 行为：把今天日程发到 Telegram (Bot Family)

- 删除：随时可在 设置 → 定时任务 删除"

→ 用户点 [允许一次] / [允许这个目标永久] / [拒绝]

→ 允许后：cronjob 实际写入 ~/.hermes/cron/jobs.json

→ 触发时直接执行，不再弹窗

[Power User]

agent → cronjob create → 直接执行

### 4.2 即时发送审批（Messaging Tool 在 /chat 中直接调用）

用户说"把刚才聊的总结发到 #engineering"，agent 调用 `send_message`：

[Standard User]

agent → send_message

→ 拦截，弹 Tauri 审批对话框

→ "AI 想发送消息：

- 目标：飞书 #engineering

- 内容预览：...（前 500 字）

- 附件：1 张图（缩略图预览）"

→ [允许一次] / [允许此目标本会话内免审批] / [拒绝]

[Power User]

agent → send_message → 直接发送

### 4.3 审批策略的"白名单豁免"

为了不让审批变成骚扰，加几个不弹窗的情况：

| 场景                              | 是否弹窗                | 理由                    |
| ------------------------------- | ------------------- | --------------------- |
| 发到 `desktop` 平台（本地交付）           | ❌ 不弹                | 没出本机，不存在误发风险          |
| 同会话内同一目标，30 分钟内重复发送             | ❌ 不弹（如果用户勾过"本会话允许"） | 体验                    |
| Cron 触发的 send_message（已在创建时审批过） | ❌ 不弹                | 防止后台时无法响应             |
| Power user 模式                   | ❌ 全部不弹              | 用户主动加 power user 视为知情 |
| 其他所有外发                          | ✅ 弹                 | 默认安全                  |

## 六、本地交付（"desktop" 平台）的设计

这是上游没有的东西，是 Kabuqina 的延伸。两种实现思路：

方案 A：在 Hermes 平台体系里注册 "desktop" 平台

- 在 `gateway/platforms/desktop.py` 写一个新 adapter
- `send_message(target="desktop", ...)` 时进入这个 adapter
- adapter 把消息 POST 给 Tauri Rust 的 loopback endpoint
- Tauri 推 Windows 通知 + 写入 /chat 消息流
- 优点：与现有协议完美对齐，cron `delivery_target="desktop"` 即可
- 缺点：要走 platform_registry 注册流程

方案 B：在 cron/scheduler 加一个 "no target = desktop" 短路

- 不动 send_message 链路
- cron job 没设 `delivery_target` 时直接调 Tauri endpoint
- 优点：改动小
- 缺点：偏离上游协议，messaging tool 用 `target="desktop"` 时就不走这条路

推荐 A，统一性高。`desktop` 也是天然作为 cron 默认目标的最佳选择。

## 七、对你提出的策略的精确实现

> "放到白名单，power-user 不用审批，standard user 加审批"

具体翻译：

| 工具                             | 白名单           | Standard User | Power User |
| ------------------------------ | ------------- | ------------- | ---------- |
| `cronjob` create/update/delete | ✅ 在 KEEP_LIST | 弹审批（创建时一次）    | 直接执行       |
| `cronjob` list/run             | ✅ 在 KEEP_LIST | 不弹（只读 + 手动触发） | 直接执行       |
| `send_message` to remote       | ✅ 在 KEEP_LIST | 弹审批（每次/每会话）   | 直接执行       |
| `send_message` to `desktop`    | ✅ 在 KEEP_LIST | 不弹（本地交付）      | 不弹         |

## 八、风险 & 必须先决定的事

在动手前需要你拍板这几个：

Q1. Standard user 删除 cron 任务要不要审批？

-  不审批（只有创建敏感），agent 帮用户清理用户自己创建的任务，弹窗反而烦。

Q2. Cron job 的 `delivery_target` 默认值？

- 默认 `desktop`（本地交付），大多数提醒类场景"提醒在桌面"就够了，发远程是少数。

Q3. Power user 切回 standard user 时已存在的 cron 任务怎么办？

- 只暂停发到远程渠道的，本地的继续运行。

Q4. Cron ticker 的运行位置？

- Web child 启动一个 60s 周期线程，已经有文件锁防重入，最简单。代价是 web child 重启时会有 ≤60s 抖动。

Q5. "本会话允许此目标永久" 这个豁免开关，是否持久化到下次启动？

- 仅本会话，重启失效（最安全），messaging 是高敏行为，每次启动重新审批是合理的安全姿态。
