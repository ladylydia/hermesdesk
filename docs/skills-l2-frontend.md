# L2 Skills：前端展示策略（设计备忘）

> 依据 [`docs/skills-design-decision.md`](skills-design-decision.md) 中的分层（L1 使用 / L2 安装官方签名 / L3 高阶来源与创作），本文只讨论 **L2 在前端如何落地**：信息架构、页面职责、与 L1/L3 的边界，以及和现有 Hermes 控制台能力的关系。  
> **状态**：产品与 IA 对齐用说明文档；实现时需对齐上游 API 与 manifest。

---

## 1. L2 在前端要解决的问题

| 层级 | 用户心智 | 前端要交付的**核心** |
|------|----------|----------------------|
| **L1** | 不必知道「Skill」这个词 | 快捷入口（快捷指令 / Recipes），偏**触发**，安全由内置白名单保证。 |
| **L2** | 发现、理解、**放心安装**「官方/合作方已签名」的扩展 | **信任展示 + 能力边界 + 安装决策**；不是仅把 `getSkills()` 做成一张表。 |
| **L3** | 高阶与创作 | 未签名来源、编辑器等，与 L2 **分轨展示**，强警告。 |

L2 的差异化在于：**在默认安全模型下**，让用户能完成「发现 → 读明白会动哪些资源 → 显式同意 → 安装/升级/管理」的闭环。  
因此 L2 的前端主角是：**发现层 + 详情/授权（Permission sheet）+ 已安装管理**，而不仅是开关列表。

---

## 2. 信息架构：主战场放在哪

1. **主战场：Hermes 控制台（`hermes/web` 等 dashboard）**  
   设计决策已明确：L2 呈现为控制台里的 **「Recipe Book / 配方书」** 类体验，与**市场、签名、版本、依赖、出网/文件范围**等长文案、多列布局匹配。与对话主界面抢同一屏会严重挤占认知。

2. **壳内（`web`，Tauri 壳 + `/chat`）：只做薄引导（可选）**  
   例如：当「技能市场相关提示」开关打开时，用**一句**说明如何进入控制台中的配方/技能入口；**不在壳内复制完整市场**。

3. **对用户用语**  
   面向用户优先：**「配方书 / Recipe Book」「已签名配方」** 等；**L2** 留在文档与开发者设置中即可。

---

## 3. 单页内建议拆成四条链路（L2 核心 UI）

| 区块 | 职责 | 为何是 L2「核心」 |
|------|------|-------------------|
| **发现** | 搜索、分类、标签、**官方/已签名**筛选；区分「可安装 / 已安装 / 有更新」 | 没有则 L2 只是运维列表，没有「市场」价值。 |
| **详情（安装前必看）** | 摘要、作者、版本、**签名状态**（如 HermesDesk 官方 / 合作方）、**由 manifest 驱动的能力说明**（读哪些目录、出网到哪些域、是否含定时等） | 与「随便装插件」划界；需与 `SKILL.md` 等声明对齐，**不应靠扫代码猜权限**。 |
| **安装 / 升级（Permission sheet）** | 安装或升级时走**固定版式**的确认页（类似 Chrome 扩展安装前的权限说明）：**取消** / **安装** 明确。 | 这是 L2 **信任** 的落点，也是合规感来源。 |
| **已安装管理** | 启用/禁用、更新、卸载；与现有 Skills 列表 + 开关**衔接**，但需区分**来源**（市场 vs 内置 L1 等） | 避免用户以为只是「多几个开关」。 |

与现状的关系：Hermes 侧 **Skills 列表 + Switch** 已具备 **「已安装管理」** 骨架；L2 要补的主要是 **市场发现流** 与 **安装前 manifest/权限的专门展示**，而不是再平行做一套无差异页面。

---

## 4. 与 L1、L3 的边界（避免前端打架）

- **L1**  
  - 壳或对话旁：**小入口**「快捷指令 / Recipes」即可。  
  - 尽量**不**向普通用户强推「Skill」一词（与设计文档一致）。

- **L2**  
  - **「市场/配方书」入口**在打开「显示技能市场」等**高级开关**后可见（与「仅多一个 UI、不扩权」的设定一致）。  
  - 与 L1 的 UI **分层**：L1 是**一键动作**；L2 是**可安装包 + 信任与版本**。

- **L3**  
  - 可共用 **Recipe Book** 的壳，但 **「更多来源」**（未签名、本地路径、编辑器等）必须**独立分区 + 强警告样式**，**不可**与 L2 官方/签名流混在同一视觉层级。

---

## 5. 与「对话（Chat）」的关系

- L2 **不必**在聊天里做完整市场；避免把聊天页变成应用商店。  
- 推荐：安装与授权仍在 **控制台** 完成；对话里最多做 **轻桥接**（例如提示「某配方已启用」或将来若有「从对话选配方跑一步」的显式操作）。  
- 这样 L2 的 **核心仍落在「可控安装 + 可信 UI」**，而不是在对话流里完成复杂授权。

---

## 6. 可执行的落地顺序（建议）

1. **产品/技术对齐**（阻塞前端细化）  
   - 市场列表与详情：**API 形态**（是否分页、是否含签名元数据、与 Hermes 现有 `getSkills` 的差异）。  
   - **Permission sheet** 的字段：是否 **100% 由 manifest 生成**，还是允许后端附加一行**人类可读摘要**（需写清规则，避免和 manifest 矛盾）。

2. **Hermes 控制台**  
   - 在现有 **Skills/配方** 导航下扩展：**发现（市场）/ 已安装** 的 IA，补 **详情页 + 安装/升级 sheet** 的线框与文案槽位。  
   - 已安装列表与现有 **Switch** 行为对齐，并 **标注来源**（市场 vs 内置）。

3. **壳**  
   - 仅保留**短引导**与设置项联动（若已有「技能市场提示」等开关，保持「轻、不抢戏」）。

---

## 7. 一句话给排期

**L2 要在前端单飞一条「可发现 + 可理解 + 可同意 + 可管理」链路；主界面在 Hermes 控制台的 Recipe Book/Skills 体系上扩展，壳只承载 L1 与轻引导。**
